# FIX-284 测试用例

## 问题描述

### 问题1：Windows 中文命令乱码
在 Windows 上，`execute_command` 工具调用含有中文字符的命令时，`cmd /c` 收到的中文字符乱码，导致命令执行失败。

### 问题2：乱码导致的失败被误报为超时
在 Windows 上，乱码导致命令立即返回 exit code 1 的 ExitError，但 `isSignaledExit()` 对 Windows 上的所有 ExitError 均返回 true，导致被误判为"命令超时"。

---

## 测试用例 1：中文目录列表（验证命令编码）

### 前置条件
- 系统: Windows
- 存在包含中文字符的目录（如 `F:\agent\life\research\既往承保意见工具资料`）

### 测试步骤
1. LLM 调用 `execute_command` 执行 `cmd /c "cd /d F:\agent\life\research\既往承保意见工具资料 && dir /b"`
2. 观察执行结果

### 预期结果
- 命令成功执行，返回目录中的文件列表
- 输出中文字符显示正常，无乱码

### 验证方法
- 在 Windows 上运行，观察输出是否为正确的中文文件列表，而非 `б` 乱码

---

## 测试用例 2：中文命令失败后错误消息（验证超时误报修复）

### 前置条件
- 系统: Windows

### 测试步骤
1. LLM 调用 `execute_command` 执行一个包含中文但路径不存在的命令（如 `dir 不存在的目录`）
2. 观察执行结果

### 预期结果
- 命令立即返回错误（exit code 1）
- 错误消息为 `command failed` 而不是 `command timed out after N seconds`

### 验证方法
- 在 Windows 上运行，观察错误消息是否包含 `timed out` 字样

## 测试用例 3：Unix 平台不引入回归问题

### 前置条件
- 系统: Linux/macOS

### 测试步骤
1. 执行包含英文的正常系统命令
2. 执行一个会超时的命令（如 `sleep 10`，设 timeout=3）

### 预期结果
- 步骤1：命令正常执行，输出正确
- 步骤2：命令在超时后返回 `command timed out after 3 seconds`

### 验证方法
- 在 Unix 上运行，观察行为与修改前一致
