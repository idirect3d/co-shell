# co-shell 项目规则

## 项目概述
co-shell 是一个智能命令行 Shell，通过自然语言与 AI Agent 交互，智能编排和执行系统命令。支持 MCP 协议、记忆管理、上下文管理和全局规则。

## 作者信息
- 作者: L.Shuang
- 仓库: https://github.com/idirect3d/co-shell
- 所有源文件头部必须包含作者信息、创建日期、最后修改日期以及版权声明（MIT）

## 技术栈
- 语言: Go 1.22+
- REPL: github.com/c-bata/go-prompt
- LLM: github.com/sashabaranov/go-openai（兼容 OpenAI/DeepSeek/国产模型）
- MCP: github.com/mark3labs/mcp-go
- 存储: go.etcd.io/bbolt（嵌入式 KV 数据库）

## 架构约束

### 目录结构
- `main.go` — 程序入口，初始化所有模块
- `config/` — 配置管理（LLM/MCP/Rules），JSON 持久化到 ~/.co-shell/
- `repl/` — REPL 交互层，使用 go-prompt，Tab 补全
- `agent/` — Agent 核心循环：LLM 调用 → 工具执行 → 迭代
- `llm/` — LLM 客户端抽象，OpenAI 兼容 API，流式支持
- `mcp/` — MCP 客户端管理器，多 Server 连接、工具调用
- `store/` — 持久化存储，bbolt 实现记忆/上下文/会话
- `cmd/` — 内置命令处理器（settings/mcp/rule/memory/context）

### 内置命令命名规范
所有内置命令以 `.` 开头，统一放在 `cmd/` 目录：
- `.settings` — LLM API 参数管理
- `.mcp` — MCP Server 管理
- `.rule` — 全局规则管理
- `.memory` — 记忆管理
- `.context` — 上下文管理

### 接口定义规范
- LLM 客户端接口定义在 `llm/client.go` 中
- MCP 管理器接口定义在 `mcp/client.go` 中
- Store 接口定义在 `store/bolt.go` 中
- 所有接口使用 `interface{}` 类型定义，实现类不对外暴露

## 编码规范

### 通用规则
- 所有错误必须显式处理，不允许使用 `_` 忽略错误
- 错误信息使用 `fmt.Errorf("context: %w", err)` 模式包装
- 函数参数超过 3 个时使用结构体或 Option 模式
- 使用 `context.Context` 作为第一个参数传递超时和取消信号
- 所有导出函数必须有 GoDoc 注释

### 命名规范
- 包名使用小写单数形式（如 `config`、`store`）
- 接口名使用 `-er` 后缀（如 `Client`、`Handler`）
- 私有方法使用驼峰命名
- 常量使用驼峰命名（Go 风格，非全大写）

### 并发安全
- MCP Manager 使用 `sync.RWMutex` 保护共享状态
- Agent 循环使用 `context.Context` 控制超时
- 所有可关闭资源实现 `Close() error` 方法

### 测试要求
- 测试文件放在同目录下，命名 `*_test.go`
- 使用 table-driven tests 模式
- 关键逻辑（Agent Loop、命令解析）必须有单元测试

## 依赖管理
- 不要随意修改 go.mod 中的依赖版本
- 新增依赖需评估必要性，优先使用标准库，高度审慎引入第三方库
- 绝对禁止引入个人的、来源不明的或已经过时的第三方库
- 尽全力降低对第三方库的依赖
- 避免引入重量级框架（如 gin、cobra）

## 安全约束
- 系统命令执行必须设置超时（默认 30 秒）
- API Key 等敏感信息显示时需脱敏（显示前 4 位 + **** + 后 4 位）
- 所有文件操作需先确认再执行
- MCP Server 命令执行需验证参数合法性

## 语言偏好
- 代码注释使用英文
- 用户可见的提示信息使用中文
- 变量命名使用英文

## 开发工作流
1. 先在 `cmd/` 中添加命令处理器
2. 在 `agent/` 中注册新工具
3. 在 `repl/` 中添加 Tab 补全
4. 运行 `go build ./...` 验证编译
5. 运行 `go vet ./...` 检查代码质量
6. 在 ROADMAP.md 中标记已完成的任务，并标注当前 BUILD 编号
7. 完成任务后，将当前功能分支合并到主分支（main），并推送
8. 完成一个版本后，使用版本号打 Git 标签（如 `v0.1.0`、`v0.2.0`），并推送标签



## 新任务管理
- 如果用户提交的任务在 ROADMAP.md 中不存在，必须在开始开发前，在 ROADMAP.md 当前正在开发的版本（如 v0.1.0）的功能清单中追加该任务的描述
- 新任务需分配一个唯一的编号（格式：`FEATURE-XXXXX` / `ENHANCEMENT-XXXXX` / `FIX-XXXXX`），编号接续 ROADMAP 中已有的最大编号
- 在 ROADMAP.md 中记录任务后，从主分支（main）创建一个以任务编号为名称的分支（如 `FEATURE-00052`），然后在该分支上进行开发


## 已有任务修改规则
- 如果修改的内容是为了完善 ROADMAP.md 中正在修改的当前版本已有的经验证实际尚未完成的任务（如修复实现缺陷、补充功能），则直接更新该任务的描述和 BUILD 编号，不新增任务条目
- 更新描述时保留原有编号，仅更新 BUILD 编号和描述内容（如必要）

## BUILD 计数器机制
- ROADMAP.md 顶部维护一个 BUILD 计数器，格式为 `当前 BUILD: XXXXX`
- 每次 `go build ./...` 编译成功后，BUILD 编号 +1
- 完成任务时，在任务后标注 `[BUILD-XXXXX]` 标记完成时的编译版本
- BUILD 编号从 00001 开始，5 位数字，不足补零

## ROADMAP 保密规则
- ROADMAP.md 包含内部开发计划和未完成功能，**不得公开**到 GitHub 仓库
- README.md 中的版本历史仅列出已发布版本的已实现功能，不引用 ROADMAP.md
- 任何公开文档（README.md、USAGE.md 等）不得提及 ROADMAP.md 中的未完成功能

## 输出日志
1. 在运行co-shell命令当前文件夹下生成日志文件，记录必要的信息，如内部错误信息，调试信息等，可通过配置关闭。
